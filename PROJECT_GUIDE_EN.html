01. WHAT IS THIS PROJECT? Overview
/ ANALOGY /

Hundreds of academic papers are published online every single day. Nobody has time to read them all. PaperFeed lets you define "your research interests", then automatically hunts down relevant papers, scores them, and presents them on a clean website.

Think of it as a personal research assistant that reads thousands of papers every day and surfaces only the ones that match your interests — ranked by relevance.

Input

Your research topic, specific questions, and keywords of interest

e.g., "Latest papers on AI explainability"

Output

A ranked list of relevant papers displayed on a filterable web page

e.g., 23 relevant papers surfaced from 127 candidates

3 Paper Sources

arXiv, Semantic Scholar, and OpenAlex — searched simultaneously

No AI Scoring

Relevance is calculated with TF-IDF math — reproducible and fully transparent

No Server Needed

Python collects data → generates static files → browser reads them directly

03. STEP-BY-STEP PIPELINE How papers are processed
/ ANALOGY /

The pipeline works like a factory conveyor belt. Each paper passes through one stage at a time — getting processed, scored, filtered, and finally published to the website.

01
Load Config Configuration Loading config.py
Reads data/survey_config.json.
This file holds your research topic, search keywords, and the minimum score threshold for a paper to be accepted.
INPUT: survey_config.json + .env (environment variables)
OUTPUT: SurveyConfig object (a bundle of all settings)
02
Fetch Papers Multi-source Crawling fetcher.py
Sends requests to all 3 paper databases simultaneously and collects results.

arXiv — Free preprint repository, cutting-edge papers
Semantic Scholar — AI/ML-focused academic database
OpenAlex — The broadest open academic database

Up to 50 papers per source, yielding ~150 "candidate" papers total.
INPUT: Search queries (keyword list)
OUTPUT: ~150 raw paper candidates
03
Deduplicate Remove Duplicates deduplicator.py
The same paper can appear in all 3 sources. This step merges duplicates and keeps the most complete record.

Identity Priority: DOI ▶ arXiv ID ▶ OpenAlex ID ▶ S2 ID ▶ Title fingerprint
(DOI is the most trustworthy unique identifier)
INPUT: ~150 papers (with duplicates)
OUTPUT: ~80–100 unique papers
04
Score Papers TF-IDF Relevance Scoring scorer.py
Mathematically measures how similar each paper's title + abstract is to your research topic.

TF-IDF + Cosine Similarity: Converts text into vectors based on word frequency, then measures the angle between them. Outputs a score from 0.0 to 1.0.
No LLM (like ChatGPT) involved — same inputs always produce the same score.
INPUT: Deduplicated papers
OUTPUT: Each paper gets a score field (e.g., 0.73)
05
Filter Accept / Reject run.py
Papers that meet the minimum score threshold are ACCEPTED.
Papers that don't are REJECTED — with a reason logged.

Rejection reasons: score too low / outside year range / missing title or URL
INPUT: Scored papers
OUTPUT: accepted list + rejected list (with reject_reason field)
06
Save & Publish Persist Results publisher.py
Saves results to disk and updates the website data.

— Appends new papers to data/papers.ndjson
— Appends rejected papers to data/rejects.ndjson
— Records run metadata in data/run_history.ndjson
— Mirrors everything as JSON into site/data/
INPUT: accepted + rejected paper lists
OUTPUT: Updated files → website auto-refreshes
07
Done Status → idle
Status flips to "idle". The web frontend automatically reloads the new data within 3 seconds.

06. CONFIGURATION Tuning the pipeline
/ KEY CONFIG FILE: data/survey_config.json /

This is the only file you need to edit to control what gets searched and how strictly papers are filtered. 

{
  "topic_overview": "artificial intelligence",
  // ↑ Main research topic. One sentence describing the field.

  "research_questions": [
    "What methods are used to make AI systems explainable?",
    "How can AI be applied to healthcare diagnosis?"
  ],
  // ↑ Specific questions you want answered. More questions = more precise search.

  "query_hints": ["XAI", "explainability", "interpretability"],
  // ↑ Extra keywords. Acronyms and synonyms work great here.

  "timeline_from_year": 2020,
  "timeline_to_year": 2026,
  // ↑ Papers outside this year range are automatically rejected.

  "min_relevance_score": 0.05
  // ↑ Minimum relevance score. 0.05 = 5%. Higher = stricter. Range: 0.0–1.0
}
/ ENVIRONMENT VARIABLES (.env file) /
S2_API_KEY=your_api_key_here
# Semantic Scholar API key (optional but recommended for more results)
# Get a free key at: semanticscholar.org/product/api

DAYS_BACK=7
# How many days back to search for papers (default: 7)

MAX_PER_SOURCE=50
# Max papers to fetch per source (default: 50)
/ min_relevance_score GUIDE /
ValueStrictnessWhen to use
0.03 – 0.05 Very low (permissive) Too few results showing up; early exploration of a topic
0.10 – 0.20 Moderate General use — a good starting point (recommended)
0.30 – 0.50 High (strict) Very specific topic; prioritize precision over recall
07. HOW TO RUN Getting started
/ 1. One-time setup /
# Install dependencies
pip install -r requirements.txt

# Create your .env file
cp .env.example .env
# Open .env and add your API key
/ 2. Run the pipeline /
# Default run
python -m app.run

# Override topic temporarily
python -m app.run --topic "robotics"

# Fetch last 14 days of papers
python -m app.run --days-back 14

# Dry run — simulate without saving
python -m app.run --dry-run
/ 3. View the website /

08. DEBUGGING GUIDE When things go wrong
/ COMMON ISSUES & FIXES /

Work through this checklist in order when something isn't working.

Issue 1: Zero papers accepted (0 papers accepted)
Cause: min_relevance_score is set too high, or search queries are too narrow
Fix:
1. Lower min_relevance_score to 0.05 in data/survey_config.json
2. Run with --dry-run and check scorer.py output (min/max/mean scores printed)
3. Add broader keywords to query_hints
Issue 2: Semantic Scholar error (429 Too Many Requests)
Cause: Rate limit exceeded — running without an API key
Fix:
1. Get a free API key at semanticscholar.org/product/api
2. Add S2_API_KEY=your_key_here to your .env file
3. The pipeline still works without a key, but returns fewer results
Issue 3: Website not updating (papers not appearing)
Cause: site/data/ JSON files weren't regenerated, or browser cache
Fix:
1. Confirm python -m app.run completed successfully
2. Check the last_run time in data/system_status.json
3. Hard-refresh the browser (Ctrl+Shift+R / Cmd+Shift+R)
4. Verify that papers.json exists in site/data/
Issue 4: ModuleNotFoundError
Cause: Python dependencies not installed
Fix: Run pip install -r requirements.txt
Required packages: httpx, feedparser, scikit-learn, python-dateutil
Issue 5: Duplicate papers appearing on the website
Cause: Paper has no DOI or arXiv ID — title fingerprint dedup failed
Fix:
1. Manually inspect paper_id values in data/papers.ndjson
2. Check publisher.py's existing_ids set for cross-run duplicates
3. This is rare for arXiv papers (they always have a clear arXiv ID)
Issue 6: CORS error in browser (when opening the file directly)
Cause: Opening index.html via file:// protocol blocks JSON loading
Fix: Always access the site through a local web server:
cd site && python -m http.server 8080
→ Open http://localhost:8080 in your browser
/ DEBUGGING CHECKLIST (run in order) /
□ 1. Check .env file exists
   ls -la .env

□ 2. Validate survey_config.json syntax
   python -c "import json; json.load(open('data/survey_config.json'))"

□ 3. Check last pipeline run status
   cat data/system_status.json

□ 4. Count accepted papers
   wc -l data/papers.ndjson

□ 5. Check most recent run record
   tail -1 data/run_history.ndjson

□ 6. Test run (no files written)
   python -m app.run --dry-run

□ 7. Confirm site/data/ files exist
   ls -la site/data/
