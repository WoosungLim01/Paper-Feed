01. 이 프로젝트가 뭐하는 건가요? What is this?
/ 쉬운 비유 (Analogy) /

인터넷에는 매일 수백 편의 학술 논문이 쏟아집니다.
그걸 일일이 읽을 수는 없죠. PaperFeed는 "내가 관심있는 주제"를 알려주면, 자동으로 논문을 찾아다가 점수를 매겨서 웹사이트에 정리해주는 시스템입니다.

Like a personal research assistant who reads thousands of papers every day and presents only the ones relevant to your interests.

Input (입력)

관심 있는 연구 주제, 궁금한 질문들, 키워드

e.g., "AI의 설명 가능성에 관한 최신 논문"

Output (출력)

관련도 점수 순으로 정렬된 논문 목록이 담긴 웹페이지

e.g., 127개 논문 중 관련 논문 23개 표시

논문 소스 3개

arXiv, Semantic Scholar, OpenAlex — 3곳에서 동시에 검색

AI 없이 점수 계산

TF-IDF 수학 알고리즘으로 관련도 계산. 재현 가능하고 투명함

서버 불필요

Python으로 데이터 수집 → 정적 웹사이트 생성 → 브라우저에서 보기

02. 어떻게 작동하나요? Big Picture
/ 전체 구조 (Architecture Overview) /

이 시스템은 크게 3개의 레이어(층)로 나뉩니다. Python 백엔드가 논문을 수집/분석하고, 데이터 파일로 저장하고, 웹 프론트엔드가 그걸 예쁘게 보여줍니다.

LAYER 1 — Python Backend (백엔드)
FETCHER
논문 수집
3개 API에서 크롤링
SCORER
관련도 점수
TF-IDF 알고리즘
PUBLISHER
결과 저장
파일로 출력
▼
LAYER 2 — Data Files (데이터)
papers.ndjson
채택된 논문
관련도 통과한 논문
rejects.ndjson
탈락된 논문
관련도 미달 논문
run_history.ndjson
실행 기록
언제 얼마나 찾았나
▼
LAYER 3 — Web Frontend (프론트엔드)
site/app.js (React)
논문 뷰어
검색, 필터, 즐겨찾기
site/data/*.json
미러된 데이터
브라우저가 읽는 JSON
03. 파이프라인 단계별 설명 Step-by-Step Pipeline
/ 비유 /

파이프라인(Pipeline)은 컨베이어 벨트와 같습니다. 논문이 한 단계씩 통과하면서 가공되고, 최종적으로 웹사이트에 게시됩니다.

01
설정 불러오기 Load Config config.py
data/survey_config.json 파일을 읽습니다.
여기에 "내가 관심있는 주제", "찾고 싶은 키워드", "몇 점 이상만 보여줄까" 같은 설정이 들어있습니다.
INPUT: survey_config.json + .env (환경변수)
OUTPUT: SurveyConfig 객체 (설정값 묶음)
02
논문 수집 (Fetch) Fetching Papers fetcher.py
3개의 논문 데이터베이스 API에 동시에 요청을 보내 논문을 긁어옵니다.

arXiv — 프리프린트 논문 저장소 (무료, 최신)
Semantic Scholar — AI/ML 논문 특화 DB
OpenAlex — 가장 넓은 범위의 학술 논문 DB

소스당 최대 50개, 총 최대 150개의 "후보" 논문을 가져옵니다.
INPUT: 검색 쿼리 (키워드 목록)
OUTPUT: ~150개의 논문 후보 (원본 데이터)
03
중복 제거 (Deduplicate) Deduplication deduplicator.py
같은 논문이 3개 소스에서 각각 발견될 수 있습니다.
이 단계에서 중복을 합치고, 가장 완전한 정보만 남깁니다.

우선순위: DOI ▶ arXiv ID ▶ OpenAlex ID ▶ S2 ID ▶ 제목 지문
(DOI가 가장 믿을 수 있는 식별자)
INPUT: ~150개 (중복 포함)
OUTPUT: ~80-100개 (중복 제거된 유니크 논문)
04
점수 계산 (Score) Scoring with TF-IDF scorer.py
각 논문의 제목+초록이 내 관심 주제와 얼마나 비슷한지 수학적으로 계산합니다.

TF-IDF + Cosine Similarity 알고리즘: "단어 빈도"를 기반으로 텍스트 유사도를 0.0 ~ 1.0 사이 점수로 환산합니다.
LLM (ChatGPT 같은 AI)을 사용하지 않아서 항상 같은 결과가 나옵니다.
INPUT: 중복 제거된 논문들
OUTPUT: 각 논문에 score 필드 추가 (예: 0.73)
05
필터링 (Filter) Accept / Reject run.py
설정된 최소 점수 기준을 넘는 논문은 채택(ACCEPT),
못 넘는 논문은 탈락(REJECT)으로 분류됩니다.

탈락 이유: 점수 미달 / 연도 범위 밖 / 제목·URL 없음
INPUT: 점수가 붙은 논문들
OUTPUT: accepted 목록 + rejected 목록 (이유 포함)
06
저장 & 게시 (Publish) Save & Publish publisher.py
결과를 파일로 저장하고 웹사이트 데이터를 업데이트합니다.

— data/papers.ndjson에 새 논문 추가
— data/rejects.ndjson에 탈락 논문 추가
— data/run_history.ndjson에 이번 실행 기록 저장
— site/data/*.json에 웹용 JSON 복사
INPUT: accepted + rejected 논문 목록
OUTPUT: 업데이트된 파일들 → 웹사이트 자동 갱신
07
완료 (Done) Status → idle
상태가 "idle"로 바뀌고, 웹 프론트엔드가 3초 후 자동으로 새 데이터를 불러옵니다.
/ STATUS PROGRESSION (파이프라인 진행 상태) /
crawling
→
deduplicating
→
scoring
→
filtering
→
publishing
→
idle

웹사이트 우측 상단의 상태 카드(Status Card)에서 이 진행 상황을 실시간으로 볼 수 있습니다.

04. 파일 구조 File Structure
/ 비유 /

프로젝트 폴더를 "사무실"에 비유하면: app/은 직원들(각 역할을 하는 Python 코드), data/는 캐비닛(데이터 파일), site/는 진열장(웹사이트)입니다. 

paper-discovery/
│
├── app/                    ← Python 백엔드 (핵심 엔진)
│   ├── run.py              ← 총괄 지휘자 (파이프라인 실행)
│   ├── config.py           ← 설정 로더
│   ├── fetcher.py          ← 논문 수집기 (3개 API)
│   ├── deduplicator.py     ← 중복 제거기
│   ├── scorer.py           ← TF-IDF 점수 계산기
│   ├── store.py            ← 파일 읽기/쓰기 도우미
│   └── publisher.py        ← 결과 저장 & 웹사이트 업데이트
│
├── data/                   ← 데이터 저장소 (append-only)
│   ├── survey_config.json   ← ⭐ 핵심 설정파일 (주제, 질문, 임계값)
│   ├── papers.ndjson        ← 채택된 논문들 (줄마다 JSON 1개)
│   ├── rejects.ndjson       ← 탈락된 논문들 (이유 포함)
│   ├── run_history.ndjson   ← 실행 기록들
│   ├── changelog.md         ← 사람이 읽기 좋은 변경 로그
│   └── system_status.json   ← 현재 파이프라인 상태
│
├── site/                   ← 웹 프론트엔드 (React)
│   ├── index.html           ← HTML 껍데기 (React 마운트 포인트)
│   ├── app.js               ← React 앱 (696줄 — 모든 UI 로직)
│   ├── styles.css           ← 스타일 & 애니메이션
│   └── data/               ← Python이 생성한 JSON 미러
│       ├── papers.json
│       ├── rejects.json
│       ├── run_history.json
│       └── system_status.json
│
├── requirements.txt         ← Python 패키지 목록
├── .env                     ← 비밀 API 키 (git에 올리지 않음)
└── .env.example             ← .env 템플릿

/ 논문 데이터 구조 (Paper Object Fields) /
식별자 (IDs)

paper_id — 내부 고유 ID
arxiv_id — arXiv 논문 번호
doi — 국제 표준 논문 번호
openalex_id — OpenAlex ID
semantic_scholar_id — S2 ID

내용 (Content)

title — 논문 제목
abstract — 초록
authors — 저자 목록
year — 출판 연도
published_date — 날짜

메타데이터 (Meta)

source — 최초 발견 소스
source_hits — 발견된 소스 목록
score — 관련도 점수 (0~1)
run_id — 어느 실행에서 추가됐나

링크 (Links)

url — 논문 페이지 URL
pdf_url — PDF 다운로드 링크

06. 설정 방법 Configuration
/ 핵심 설정 파일: data/survey_config.json /

이 파일 하나를 수정하면 무엇을 검색할지, 얼마나 엄격하게 필터링할지 모두 바꿀 수 있습니다.

{
  "topic_overview": "artificial intelligence",
  // ↑ 메인 주제. 한 문장으로 어떤 분야인지 설명

  "research_questions": [
    "What methods are used to make AI systems explainable?",
    "How can AI be applied to healthcare diagnosis?"
  ],
  // ↑ 구체적으로 알고 싶은 질문들. 많을수록 검색이 정밀해짐

  "query_hints": ["XAI", "explainability", "interpretability"],
  // ↑ 추가 키워드. 전문 약어나 동의어를 넣으면 효과적

  "timeline_from_year": 2020,
  "timeline_to_year": 2026,
  // ↑ 이 연도 범위 밖의 논문은 자동 탈락

  "min_relevance_score": 0.05
  // ↑ 최소 관련도 점수. 0.05 = 5%. 높일수록 엄격 (0.0~1.0)
}
/ 환경변수 (.env 파일) /
S2_API_KEY=your_api_key_here
# Semantic Scholar API 키 (없어도 되지만 있으면 더 많은 결과)
# https://www.semanticscholar.org/product/api 에서 무료 발급 가능

DAYS_BACK=7
# 최근 며칠치 논문을 가져올지 (기본값: 7일)

MAX_PER_SOURCE=50
# 소스당 최대 논문 수 (기본값: 50)
/ min_relevance_score 가이드 /
값의미언제 쓸까?
0.03 ~ 0.05 매우 낮음 (관대) 관련 논문이 너무 적게 나올 때, 탐색 초기
0.10 ~ 0.20 보통 일반적인 사용 (추천)
0.30 ~ 0.50 높음 (엄격) 매우 구체적인 주제, 정밀도 우선
07. 실행 방법 How to Run
/ 1. 처음 설치 (One-time setup) /
# 패키지 설치
pip install -r requirements.txt

# .env 파일 생성
cp .env.example .env
# .env 파일 열어서 API 키 입력
/ 2. 파이프라인 실행 /
# 기본 실행
python -m app.run

# 주제 임시 변경
python -m app.run --topic "robotics"

# 최근 14일치 논문 검색
python -m app.run --days-back 14

# 테스트 (파일 저장 안 함)
python -m app.run --dry-run
/ 3. 웹사이트 보기 /
# 간단한 웹 서버 실행 (site 폴더에서)
cd site && python -m http.server 8080

# 브라우저에서 열기
# → http://localhost:8080

08. 디버깅 가이드 Debugging
/ 자주 발생하는 문제와 해결법 /

문제가 생겼을 때 이 체크리스트를 순서대로 확인하세요.

문제 1: 논문이 하나도 안 나온다 (0 papers accepted)
원인: min_relevance_score가 너무 높거나, 검색 쿼리가 너무 좁음
해결: 1. data/survey_config.json에서 min_relevance_score를 0.05로 낮추기
2. --dry-run으로 실행해서 scorer.py 출력 확인 (min/max/mean 점수)
3. query_hints에 더 일반적인 키워드 추가
문제 2: Semantic Scholar에서 에러 (429 Too Many Requests)
원인: API 속도 제한 초과 — API 키 없이 사용 중
해결: 1. semanticscholar.org에서 무료 API 키 발급
2. .env 파일에 S2_API_KEY=your_key_here 추가
3. 키 없이도 작동은 하나, 결과가 줄어들 수 있음
문제 3: 웹사이트가 업데이트 안 됨 (논문이 안 보임)
원인: site/data/ 폴더의 JSON이 갱신 안 됐거나, 브라우저 캐시
해결: 1. python -m app.run을 다시 실행했는지 확인
2. data/system_status.json의 last_run 시간 확인
3. 브라우저에서 강력 새로고침 (Ctrl+Shift+R / Cmd+Shift+R)
4. site/data/ 폴더에 papers.json이 있는지 확인
문제 4: ModuleNotFoundError (모듈 없음 에러)
원인: 의존 패키지가 설치되지 않음
해결: pip install -r requirements.txt 실행
필요 패키지: httpx, feedparser, scikit-learn, python-dateutil
문제 5: 같은 논문이 중복으로 보인다
원인: 논문에 DOI나 arXiv ID가 없어서 제목 지문으로 dedup 실패
해결: 1. data/papers.ndjson에서 paper_id 중복 여부 수동 확인
2. 이전 papers.ndjson 데이터와의 중복 → publisher.py의 existing_ids 확인
3. 대부분은 arXiv 논문에서 발생하지 않음 (arXiv ID 명확)
문제 6: 웹사이트에서 CORS 에러 (파일을 직접 열 때)
원인: file:// 프로토콜로 직접 index.html 열면 JSON 로드 불가
해결: 반드시 웹 서버를 통해 접근해야 함
cd site && python -m http.server 8080
→ http://localhost:8080 으로 접속
/ 디버깅 체크리스트 (순서대로 확인) /
□ 1. .env 파일 존재 확인
   ls -la .env

□ 2. data/survey_config.json 문법 확인
   python -c "import json; json.load(open('data/survey_config.json'))"

□ 3. 마지막 실행 상태 확인
   cat data/system_status.json

□ 4. 논문 수 확인
   wc -l data/papers.ndjson

□ 5. 가장 최근 실행 기록 확인
   tail -1 data/run_history.ndjson

□ 6. 테스트 실행 (아무것도 저장 안 함)
   python -m app.run --dry-run

□ 7. site/data/ 폴더 파일 확인
   ls -la site/data/
09. 각 파일 역할 요약 Component Summary

App Root

최상위 컴포넌트. 탭 전환(Papers/History), 다크모드 토글, 전체 데이터 로딩 담당

StatusCard

우측 상단의 파이프라인 상태 표시 (idle / crawling 등). 3초마다 갱신

PapersPage

메인 논문 목록 페이지. 사이드바 필터(소스, 연도, 점수), 검색박스, 정렬 포함

PaperCard

논문 1개를 표시하는 카드. 제목, 저자, 관련도 바, 출처 배지, 즐겨찾기 버튼

MonthGroup

논문을 출판 월별로 묶어서 접을 수 있는 섹션으로 표시

HistoryPage

실행 이력 탭. 각 실행에서 몇 개 발견했는지, 소스별 통계
